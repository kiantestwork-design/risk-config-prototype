# React 重启后测试方案

日期：2026-04-12  
适用场景：Chrome 挂死后，重启电脑后重新验证当前 React 版是否接近旧 `app.js` 版本

## 测试目标

本轮测试重点不是“页面能打开”这么简单，而是验证下面 4 类问题是否真的修好了：

1. 右侧历史版本栏是否稳定出现
2. 切换历史版本后，主编辑区是否完整同步
3. 条件表达式编辑器是否恢复到旧版风格，且不再保存坏数据
4. 编辑页整体样式是否明显接近旧 `app.js`

## 测试前准备

### 1. 重启后先清理旧进程

在项目根目录执行：

```bash
killall node || true
killall "Google Chrome" || true
```

如果不想关掉你自己的 Chrome，可以只杀开发进程：

```bash
ps -Ao pid,command | rg "vite|risk-config-prototype" 
kill <PID>
```

### 2. 启动项目

```bash
npm install
npm run dev -- --host 127.0.0.1 --port 4177
```

启动后确认地址：

```text
http://127.0.0.1:4177/
```

### 3. 建议浏览器环境

- 用一个全新的 Chrome 窗口测试
- 不要保留旧标签页自动恢复
- 最好开无痕窗口，避免缓存干扰

## 测试范围

本轮必须覆盖 4 个模块：

1. 接入点
2. 特征
3. 规则
4. 策略

每个模块都要测：

- 列表页
- 查看态
- 编辑态
- 右侧历史版本

## 一、基础可用性检查

### 1. 构建检查

执行：

```bash
npm run build
```

预期：

- 构建通过
- 允许有 chunk size warning
- 不允许有语法错误、类型错误、运行时导出错误

### 2. 首页与路由检查

手工打开以下地址：

- `/#/event-points`
- `/#/feature-list`
- `/#/rules`
- `/#/activations`

预期：

- 左侧菜单可切换
- 页面无白屏
- 控制台无红色报错

## 二、右侧历史版本专项测试

## 2.1 接入点

进入 `/#/event-points`

逐个打开：

- `EP00000001`
- `EP00000002`
- `EP00000003`
- `EP00000000`

检查项：

- 右侧是否有“历史版本”栏
- 历史版本是否至少有 1 条
- 点击不同版本后，主区域标题、描述、状态是否变化
- 接入点查看态切版本后，描述和生命周期状态是否同步变化

预期：

- 4 条接入点都能看到右栏
- 不出现“有的有右栏，有的没有”的情况

## 2.2 特征

进入 `/#/feature-list`

逐个打开：

- `user_transaction_history`
- `user_recent_trade_time_7d_rt`
- `user_login_count_1h`

检查项：

- 右侧是否都有“历史版本”
- 点击不同版本后，以下字段是否同步变化：
  - 描述
  - 类型
  - 数据来源
  - 值类型
  - 接入点
  - `includeCurrentEvent`
  - `conditionExpression`
  - `compositeKeyJsonPaths`
  - `calculationConfig`

预期：

- 3 条特征都能看到右栏
- 切历史版本后，不只是标题变化，业务字段也跟着变化

## 2.3 规则

进入 `/#/rules`

逐个打开：

- `rule_high_amount`
- `rule_freq_check`
- `rule_model_evaluation`
- `rule_new_device_login`
- `rule_amount_threshold`

检查项：

- 右侧是否都有“历史版本”
- 点击不同版本后，以下字段是否同步变化：
  - 描述
  - 条件表达式
  - 初始分
  - 基数
  - 运算符
  - 取值字段
  - 倍率
  - 上限
  - 动作配置

预期：

- 5 条规则都能看到右栏
- 不允许出现切版本后标题像另一条规则

## 2.4 策略

进入 `/#/activations`

逐个打开：

- `activation_txn_risk_45min`
- `activation_txn_amount_check`
- `activation_login_risk`
- `activation_device_fingerprint`

检查项：

- 右侧是否都有“历史版本”
- 点击不同版本后，以下字段是否同步变化：
  - 描述
  - 场景
  - 接入点
  - 优先级
  - 阈值配置
  - 关联规则

预期：

- 4 条策略都能看到右栏
- 如果 `activation_device_fingerprint` 没有右栏，说明策略版本覆盖仍然不完整

## 三、条件表达式编辑器专项测试

目标：确认表达式编辑器不只是“能显示”，而是真的接近旧版，并且不会保存坏数据。

## 3.1 视觉风格检查

打开：

- 特征：`user_recent_trade_time_7d_rt`
- 规则：`rule_high_amount`
- 规则：`rule_amount_threshold`

分别进入“编辑”态。

检查项：

- 表达式编辑器是否还是明显的“重写彩色卡片风”
- 是否更接近旧版 `app.js / Lz` 风格
- 重点看：
  - AND/OR 切换样式
  - 条件行布局
  - 添加条件入口位置
  - 添加子组入口位置
  - 删除图标位置
  - 只读态样式

预期：

- 整体风格应明显接近旧版
- 不应再像一个独立设计的新组件

## 3.2 保存正确性检查

### Case A：单层条件

在规则 `rule_high_amount` 中：

- 修改一个现有条件值
- 保存
- 重新进入该规则

预期：

- 条件值保留
- 没有结构丢失

### Case B：多组 OR 条件

在规则 `rule_amount_threshold` 中：

- 该规则已有两组条件
- 修改第二组中的一个条件值
- 保存
- 返回列表后重新进入

预期：

- 两组都还在
- 没有丢掉第二组

### Case C：新增子组

任选一条规则：

- 新增一个子组
- 在子组中新增 1 个条件
- 保存
- 返回后重新进入

预期：

- 子组仍存在
- 子组中的条件仍存在
- 不允许出现“保存后只剩当前层 conditions，groups 丢了”

### Case D：混合结构

如果 UI 支持：

- 同一组中既保留条件，又新增子组
- 保存后重新进入

预期：

- 条件和子组都保留
- 不允许只保留其中一部分

## 四、编辑页样式 parity 检查

目标：确认编辑页已经不再是明显的“统一 Tailwind 后台壳”。

分别打开：

- 一个接入点
- 一个特征
- 一个规则
- 一个策略

观察以下区域：

### 4.1 头部

检查项：

- 标题层级
- 返回按钮
- 编辑按钮 / 保存按钮 / 取消按钮
- 描述文案位置

预期：

- 更接近旧版平台编辑页
- 不是统一模板页头

### 4.2 主区分段

检查项：

- 卡片间距
- 边框颜色
- 圆角
- 阴影
- 信息分区的标题和边界

预期：

- 视觉密度接近旧版
- 不再有明显的“现代 Tailwind 演示页”感

### 4.3 右侧历史栏

检查项：

- 宽度是否稳定
- 和主编辑区的分隔是否自然
- 当前版本高亮是否清晰
- 时间、版本、编辑人信息是否层级清楚

预期：

- 右栏像旧版业务侧栏，而不是额外贴上的卡片

## 五、回归功能链路

虽然这轮重点是 parity，但还是要顺手回归两个核心流程。

### 5.1 规则编辑保存

步骤：

1. 打开一条规则
2. 进入编辑
3. 修改描述
4. 修改表达式
5. 保存
6. 返回再打开

预期：

- 描述保留
- 表达式保留
- 右侧历史栏仍存在

### 5.2 特征编辑保存

步骤：

1. 打开 `user_recent_trade_time_7d_rt`
2. 编辑
3. 修改描述或 `includeCurrentEvent`
4. 保存
5. 返回再打开

预期：

- 修改保留
- 条件表达式没有丢
- 右侧历史栏仍存在

## 六、记录方式

建议测试时按下面格式记录：

```text
[模块] 规则
[实体] rule_amount_threshold
[检查项] 历史版本切换
[结果] 失败
[现象] 切到 v1 后第二组 OR 条件消失
[截图] xxx.png
```

至少记录：

- 模块
- 实体
- 检查项
- 结果
- 现象
- 是否可复现

## 七、通过标准

满足以下条件才算这一轮通过：

1. 接入点 / 特征 / 规则 / 策略的指定实体都能看到右侧历史栏
2. 切换历史版本后，主编辑区关键业务字段同步变化
3. 条件表达式编辑器样式明显接近旧 `app.js`
4. 条件表达式保存后不会丢层级、丢子组、丢条件
5. 编辑页整体不再像“统一 React 模板后台”
6. `npm run build` 通过

## 八、如果测试失败，优先级怎么排

按下面顺序修：

1. 历史栏缺失
2. 历史版本切换后字段不同步
3. 表达式保存坏数据
4. 表达式样式不像旧版
5. 编辑页整体样式差异
